# QuickNote 同步与图片架构

本文档描述 `fix/sync-hardening` 分支引入的同步状态机、账户边界和图片缓存策略，作为后续维护与评审基线。

## 1. 不变量

### 账户隔离

- 所有笔记、待办、清单、附件 Blob 和待同步操作必须带 `ownerKey`。
- 所有查询、写入、同步重放、清理和统计必须限定到当前 `homeAccountId`。
- 账户切换会推进 owner generation；旧账户同步结果不得更新当前账户的全局状态。
- Delta 游标和同步状态按账户存储。

### 本地写入与 Outbox

- 本地实体更新与待同步操作必须位于同一个 Dexie 事务。
- 每个实体维护 `localRevision` 和 `syncedRevision`。
- 同步请求捕获 `targetRevision`；响应回来后，仅当当前版本仍等于该版本时才能把实体标记为已同步。
- 队列完成采用版本比较；同步期间发生的新编辑必须保留队列。

### 远端创建与删除

- 创建便签后立即持久化远端 ID，再上传附件和回读快照；失败重试应继续更新同一个远端实体。
- Microsoft To Do 使用 linked resource 的 `externalId` 关联本地 ID；创建前先恢复可能已经成功的远端任务。
- 删除操作把远端 404 视为成功，保证重试幂等。

## 2. 同步协调器

触发条件：

- 应用启动且已登录、在线；
- Outbox 从空变为非空或实体继续修改；
- 网络恢复；
- 页面重新进入前台；
- 每 60 秒兜底检查；
- 用户手动同步。

同一账户同时只允许一个底层同步运行。单个冲突或永久失败不会阻塞其他实体和后续拉取。

队列状态：

- `pending`：等待执行；
- `retry-wait`：瞬时故障，指数退避；
- `conflict`：需要用户选择恢复策略；
- `dead-letter`：永久错误或多次失败，停止自动重试。

同步页必须展示受阻操作、最后错误和恢复按钮。

## 3. Graph 请求策略

- 所有请求带超时、`client-request-id` 和 Graph 来源校验。
- GET、PUT、PATCH、DELETE 等幂等请求对网络错误、429 和可恢复 5xx 进行退避重试。
- 非幂等 POST 不做客户端自动重试，业务层负责恢复远端身份。
- 429 优先遵循 `Retry-After`。
- To Do 列表和任务必须遍历全部 `@odata.nextLink`。
- Notes Delta 遍历全部 nextLink，只有所有页面处理完成后才提交最终 deltaLink。
- Notes 单页最多水合 6 条消息，附件最多并发下载 4 个。

## 4. 图片策略

### 下载限制

- 自动下载上限：5 MB；超过后只保存远端附件元数据。
- 用户主动加载上限：35 MB。
- 实际响应采用流式字节计数，不能只信任 `Content-Length` 或 Graph 元数据。
- 解码后像素上限：40 MP，防止高压缩比图片造成内存峰值。

### 本地存储

- 笔记表只保存附件元数据，不保存 Base64。
- 原图存入独立 `noteAttachmentBlobs` 表，使用 `Blob`。
- 列表卡片进入视口附近后才读取 Blob。
- 首次读取生成最长边 1024 px、质量 0.78 的 WebP 缩略图，存入 Cache Storage。
- 缩略图缓存最多 500 项，超过后回收至 400 项。
- 远端原图缓存预算默认为 300 MB，回收时只删除已同步且可以重新下载的远端图片，绝不删除尚未同步的本地图片。
- 用户主动查看超过 5 MB 的原图仅保留在当前页面内存中，不长期写入 IndexedDB。

## 5. 富文本安全

- Graph HTML 进入本地和渲染前都经过标签、属性和 URL 协议白名单。
- 禁止脚本、iframe、表单、SVG、事件属性和危险 URL。
- 三方合并后再次清洗并验证，无法安全合并时转为显式冲突。
- 生产部署启用 CSP、nosniff、frame 限制、权限策略和 referrer policy。

## 6. 数据迁移

- Dexie v6 为既有实体和 Outbox 补充账户作用域和 revision。
- Dexie v7 把笔记中的 Base64 附件迁移为独立 Blob 记录，并从笔记行移除二进制。
- 无法确定历史数据所属账户时使用 `__legacy__`，仅在首个没有既有数据的登录账户中认领一次，避免跨账户自动合并。

## 7. 质量门禁

每个 PR 必须通过：

- ESLint；
- TypeScript typecheck；
- Vitest；
- 生产构建；
- CodeQL。

重点故障注入用例：

- 同步 revision 1 悬挂时编辑 revision 2，旧响应不得覆盖新内容；
- A 账户离线操作不得在 B 账户重放；
- 创建请求远端成功但后续步骤失败时不得产生重复实体；
- 删除成功但响应丢失后，重试 404 应完成；
- To Do 多页数据全部保留；
- 429 遵循 Retry-After；
- 大图只保存元数据，手动下载同时受字节和像素限制。

## 8. 已知后续重构

当前 Notes Delta 已经通过 Graph 分页、单页并发上限、5 MB 自动图片阈值和图片 Blob 拆表控制峰值；最终 deltaLink 只在全部页面成功后提交。

对于拥有大量“接近 5 MB 图片”的极端首次同步账户，后续仍应把 `fetchRemoteNotesDelta` 改为 async iterator 或 page callback：每页水合后立即提交到独立 staging/session 表，全部成功后再原子发布快照与游标。实现时必须保留以下性质：

- 中途失败不推进 deltaLink；
- 重放页面幂等；
- 全量快照删除只在最后一页成功后发生；
- 旧账户 generation 失效后禁止提交；
- staging 记录可在下次启动时回收。

这项改造属于降低极端峰值的结构化优化，不应通过在现有 1293 行适配器中做未经充分测试的大范围替换来仓促完成。
